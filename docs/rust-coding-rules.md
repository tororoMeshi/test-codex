# Rust コード生成時のルール（AI向け）

このドキュメントは、Rust実装における**安全性・信頼性・可読性**のための規約です。  
AIがコードを生成・修正する際は、**必ず**本規約に従ってください。

## 1. パニック系の扱い
- Rust コードを書くとき、`unwrap` / `expect` / `panic` は原則使わないこと。
- 「本当に内部不変でしか起こりえないバグ」以外ではパニックを起こさないこと。
- どうしてもパニックが必要な場合は、その理由を**コメントで明示**し、影響範囲を限定すること。
- 例外規定は組織合意済みの範囲に限定すること。

## 2. 失敗しうる処理は Result/Option で表現
- 外部入力・I/O・ネットワーク・環境変数など失敗しうる処理は、必ず `Result` / `Option` と `?` を使って扱うこと。
- `match` / `if let` / `ok_or_else` / `map_err` などで**意味のある分岐**にすること。
- `Result<T, E>` の `E` は原則として **enum で構造化**し、文字列エラーにしないこと。

## 3. エラー型と文脈
- 必要に応じて `thiserror` / `anyhow` を使い、意味のあるエラー型と文脈付きメッセージを付けること。
  - ライブラリ・ドメイン層: `thiserror` による **明示的なエラー enum** を優先
  - アプリケーション境界（bin/handler）: `anyhow` を使って **文脈付与**してもよい
- 例:
  - `thiserror` で `DomainError` / `UsecaseError` を定義
  - `anyhow::Context` で `context("...")` を付ける

## 4. ログに依存しないエラー設計
- エラーはログに頼らず、型とメッセージから原因が追えるようにすること。
- ログは補助であり、エラー型の設計が一次情報である。

## 5. レビューコメント・修正例も同じ方針
- レビューコメント例や修正例を提示するときも、上記の方針に従うこと。

## 6. 最低限の実装スタイル
- `pub` API はドキュメントコメントを付ける（目的・入出力・失敗条件）。
- 可能な範囲で、`clippy` が指摘する危険パターンを避ける（雑な `unwrap` の温床になるため）。
- 可能な範囲で、関数は「入力→出力」が読みやすいように小さく保つ。

---

### 付録：許容される例外（限定）
以下は「内部不変」が破られたときのみ許容（ただし理由をコメントで残すこと）:
- `unreachable!()`：論理的に到達不能であることが**証明できる**場合
- `debug_assert!()`：デバッグビルドで不変条件を検証する場合
